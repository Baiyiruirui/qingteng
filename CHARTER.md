## 一、北极星目标与完成定义

**北极星**：把青藤打磨成一份"能证明 AI 应用工程能力全栈闭环"的作品集——不只是功能跑通，而是**每个 AI 环节都有评估数据背书、每个决策都有留痕**。

**完成定义（Definition of Done，投简历前必须全部满足）**：

1. 线上 Demo 全功能可用且视觉完成度高（核心页面达到登录页水准）
2. Eval 体系建成：50 题黄金集 + Langfuse 全链路 trace，README 里能贴出准确率数字
3. 四个扩展功能上线：难度自适应出题、Whisper 朗读评分、语义诗词搜索、错题专项复习

5. 暴露过的 API key 全部轮换完毕
6. `notes/talking-points/` 补齐至 015+ 篇，每个新功能都有对应讲点

3. P1、P0 不砍——如果连 P1 都保不住，说明时间盒需要重新谈，回到章程修订流程

Hono、tRPC、mem0、PWA、CN/HK 双语、教师端、闯关地图 UI、Docker、暗色模式、微信小程序。**本章程有效期内不新增任何未列入范围的功能**——新想法一律记入 `notes/` 的 backlog，下个周期再议。

## 四、各阶段工作分解与验收标准

### Phase A · 美术做透（W1-W2）

用户决策"做透再走"，据此展开：

**W1：核心页面精修**
- [ ] 诗库页 `_poems-client.tsx`：宣纸底 + 半透明卡片 + 磨砂 Header + InkField 风格搜索栏 + 空状态动效；顺手删 dead import `SealStamp`
- [ ] 聊天页 `_chat-client.tsx`：消息气泡、头部、输入区全面对标 token
- [ ] 两套颜色命名统一到 B 套短名（`paper/ink/jade/cinnabar`），删除 `qt-*`

**W2：定妆与意境图**
- [ ] 做题页、沉浸页定妆（超出"只跟 token"的精修，用户已拍板做透）
- [ ] 3 首沉浸诗意境图生成并接入（中国水墨、宣纸质感、留白、低饱和、横构图）
- [ ] 全站视觉走查：一次过完六个页面，记录并修复不一致
- [ ] 补写 `notes/talking-points/011-design-token-system.md`、`012-v0-to-production.md`

**验收**：手机 + 桌面两种视口截图走查通过；`pnpm build` 零错误；视觉上任意页面截图放进 Demo 视频都不丢人。

### Phase B · 评估与观测（W3）——所有新功能的前置

> 顾问强烈建议的顺序调整：原计划 Eval 在 Week 5，本章程提前到新功能之前。理由：C1-C3 的每一个功能都涉及 prompt 与模型调用，先有评估基线，后面每个改动都能量化回归，避免"改了 A 坏了 B"靠肉眼发现。

- [ ] Langfuse 接入：chat / opening / quiz-generate / quiz-judge / memory-extract 五条链路全部 trace（模型、token、耗时、cost）
- [ ] 50 题黄金集设计：出题质量 20 题（evidenceValid + 考点覆盖）、判题一致性 15 题（人工标注 completionRate 对照）、记忆召回 10 例、开场白质量 5 例
- [ ] Eval runner 脚本：`pnpm eval` 一键跑全集，输出通过率报告（JSON + 控制台表格）
- [ ] 跑出**第一份基线报告**，存 `notes/evals/2026-XX-baseline.md`

**验收**：任何 prompt 改动后，`pnpm eval` 十分钟内给出 diff；基线数字可写进 README。
**新增红线（写入工作约定）**：Phase B 之后，凡改动 `src/ai/prompts/` 必须跑 eval 并在 commit message 里注明结果（如 `feat: tighten judge prompt (eval 92%→96%)`）。

### Phase C1 · 难度自适应出题（W4）

- [ ] 难度模型设计：从学习画像（薄弱考点类型、近期 completionRate、错题分布）推导出题参数（考点权重 + 难度档位）
- [ ] 蓝图选点策略：优先抽学生薄弱考点，混入 20-30% 已掌握考点做巩固（间隔复习思想）
- [ ] 错题专项复习：错题本 → "专项练习"入口，只从待加强考点抽题
- [ ] Eval 扩充：自适应场景加 5-8 个测试用例
- [ ] 补写 `notes/talking-points/013-adaptive-difficulty.md`

**验收**：两个不同答题历史的账号，对同一首诗拿到明显不同的组卷；有一条 Langfuse trace 能演示"画像 → 出题参数"的推导。

### Phase C2 · Whisper 朗读评分（W5-W6 上半）

**前置条件（W4 内提前办）**：OpenAI API key 申请、R2/S3 桶开通——有外部依赖，提前一周准备，不阻塞主线。

- [ ] 录音 UI（MediaRecorder）+ 音频上传 R2
- [ ] Whisper 转写 → 与原诗逐字对齐 → 拼音相似度评分（多音字按诗中读音表处理）
- [ ] 评分反馈用青藤口吻：先肯定、指出具体字句、鼓励再读（复用判题哲学）
- [ ] events 表记录朗读事件，接入学习画像
- [ ] 补写 `notes/talking-points/014-whisper-recite-scoring.md`

**验收**：手机浏览器实测录音 → 10 秒内出分；评分对"读错一个字"敏感、对"语速停顿"宽容。

### Phase C3 · 语义诗词搜索（W6 下半）

- [ ] 每首诗"标题 + 主题 + 意象 + 首句"拼文本 → bge-m3 embedding → poems 表加 embedding 列（或独立表）+ HNSW 索引
- [ ] 搜索 API：自然语言 query → 语义召回 top-N + 结构化过滤（学段/朝代）混合检索
- [ ] 诗库页搜索栏升级：关键词命中优先、语义结果补位
- [ ] 补写 `notes/talking-points/006` 更新（从"规划中"改为实现记录）

**验收**：搜"孤独"召回《登高》《九月九日忆山东兄弟》等未出现"孤独"二字的诗；空结果率显著低于关键词搜索。

## 五、优化提纲（按模块，P0 = W7 必须关闭）

### 5.1 Memory 系统
- **P0** 记忆膨胀治理：weight × 时间衰减（如 0.9^天数）+ 每用户条数上限，超限淘汰低权重
- **P0** 沉浸模式 Memory 整合：沉浸结束后允许提取"他在《登高》里说出了自己的孤独"类记忆（当前沉浸完全不写记忆，损失最有价值的情感信号）
- **P1** 记忆查看/删除 UI（隐私边界 + 面试演示友好）
- **P1** 召回阈值 0.4 用 Phase B 的 eval 数据调优
- **P2** 画像匹配升级：作者名/别称（"老杜"）识别

### 5.2 出题与判题
- **P0** evidenceValid=false 题目的人工审核 UI（当前只入库标记，无处置闭环）
- **P0** 蓝图规模化：批量跑蓝图生成器覆盖 140 首 + 人工抽检 10%
- **P1** 判题误判率量化：黄金集中人工标注对照，目标误判方向 100% 偏"宽容"
- **P2** 组卷去重：同一学生连续两次组卷考点重合率 < 50%

### 5.3 数据与工程
- **P0** 数据清洗：dynasty 'Tang'→'唐'；全库字段一致性校验脚本
- **P0** API key 轮换（W8 硬性完成，W7 准备）
- **P1** 持久化失败重试：onFinish 写库失败进重试队列（简单方案：Redis list + 定时 flush）
- **P1** token 成本追踪：messages.meta 已有数据，聚合出每用户/每模块成本视图
- **P2** Sentry 接入（原 Week 6 计划，降级为可选）

### 5.4 体验细节
- **P1** 开场白生成失败的兜底文案（当前静默失败显示空对话）
- **P2** 沉浸模式配意境图氛围增强（依赖 Phase A 的 3 张图）
- **P2** 移动端触控细节走查
