# 0008. 协作模式:从三方循环改为 Claude Code 直接协作

**日期**: 2026-07-06
**状态**: Accepted（Supersedes 交接对话中确立的"顾问出指令 → 粘贴 → 截图 → 验收"三方循环）

## 背景

项目前半程(2026-06 中确立)用三方工作流:
- **claude.ai 顾问对话** = 长期大脑 / 产品经理,出方案、写给 Claude Code 的执行指令
- **用户** = 决策者,粘贴指令、截图反馈
- **Claude Code(VS Code 插件)** = 只管编码的"手"

这套流程当时解决的核心痛点是:**Claude Code 插件一关就失忆**,需要一个不失忆的外部大脑维持上下文。

到后半程(2026-07),该痛点已消失:CLAUDE.md + CHARTER.md + notes/ + Claude Code 跨会话记忆,新会话读一遍即归位。三方循环反而暴露出损耗。

## 决策

**日常主力改为在 Claude Code 内直接协作**;claude.ai 顾问对话降级为可选(仅碎片时间纯战略盘点用),不再由它产出"给 Claude Code 的执行指令"。

## 决策理由

1. **失忆问题已被文档+记忆体系解决**,三方循环存在的前提不再成立。
2. **顾问是"盲人"**:claude.ai 看不到代码、跑不了 build、开不了预览,只能凭上传文档+截图推理。文档描述与代码实际的偏差只有能读代码的一方能发现(例:SealStamp 被建议"保留待印猫",但印猫从未落地——见 [[0006]])。
3. **三方循环信息损耗大**:自包含指令 → 来回粘贴 → 截图汇报 → 凭截图验收,一圈下来信息掉一半,验收是"看图说话"。Claude Code 内可直接 build / 开预览截图 / push 上线,验证是真闭环。

## 保留(顾问模式里真正值钱的部分)

三方模式的价值不在"两个 AI 分工",而在**决策纪律**,这部分继续执行:
- 岔路 → 列 2-3 方案 + 编号决策点 + 标推荐项 + 用户拍板(已写入 CHARTER 第六节第 6 条)
- 大改动 → 先只读勘察 → 用户确认 → 再动手
- 每个重大决策 → 进 notes/decisions/ 留痕
- commit 规范不变(小步 commit+push、无 Co-authored-by、conventional prefix)

## 后果

**好的**:规划-拍板-执行-验证在一个闭环内完成,消除粘贴/截图损耗,决策基于代码事实而非文档推理。
**坏的**:失去"第二个 AI 独立视角"的交叉检验(用可选顾问对话弥补,但不作主力)。
**代价**:对 Claude Code 会话的决策纪律要求更高——不能因为"能直接动手"就跳过列方案/拍板环节。

## 如果反悔了

- 若发现单一 Claude Code 会话在重大战略决策上视角不够(例如产品方向大转弯),可临时拉 claude.ai 顾问做一次独立评审,但结论仍回 Claude Code 落地+留痕。
