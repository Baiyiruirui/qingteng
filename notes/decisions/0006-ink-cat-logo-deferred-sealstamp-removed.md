# 0006. 印猫 logo 暂缓落地 + 删除 SealStamp dead import

**日期**: 2026-07-06
**状态**: Accepted（Supersedes 交接对话中"保留 SealStamp 待印猫"的口头决定）

## 背景

美术阶段（CHARTER Phase A）动工前，翻查历史 claude.ai 对话（"青藤项目交接与协作启动"）时发现两处需要对齐的遗留：

1. **SealStamp dead import**：`src/app/(app)/poems/_poems-client.tsx:7` 有一个未使用的 `import { SealStamp }`。历史对话里顾问明确交代 **"先别删，阶段二马上要用印猫，这组件正好派上用场"**。
2. **印猫 logo（案 C 朱砂印猫）**：历史对话里定过 logo 从"藤字朱砂印"换成"猫住进印章里"的案 C。但该决策**从未落地**——登录页现在仍是 `<Seal char="藤" />`（`src/app/(auth)/login/page.tsx:103`）。

而 CHARTER Phase A · W1 清单与本轮执行提示词都写着"顺手删 dead import `SealStamp`"，与历史"保留"决定直接冲突。需要用户拍板。

## 决策

1. **删除** `_poems-client.tsx:7` 的 `SealStamp` dead import（在提示词 1 诗库页精修时一并做）。
2. **印猫 logo 暂缓落地**，维持现有藤字朱砂印（`<Seal char="藤" />`）不变。

## 备选方案

### 选项 A: 保留 SealStamp + 本阶段落地印猫
- 优点：一次做透，视觉身份升级到位
- 缺点：印猫是新工作量（SVG 绘制 + 定稿 + 全站替换 favicon/头像/加载态），会打断诗库/聊天页精修的连贯性；藤字印本身已够撑门面

### 选项 B: 删 SealStamp + 印猫暂缓（选中）
- 优点：范围最小、精修节奏不被打断；留着不用的 import 不干净，删掉更清爽；真要做印猫时再干净引入即可
- 缺点：印猫这个记忆点暂时缺席（但藤字印已足够）

## 决策理由

1. **章程效力最高**：CHARTER 与旧文件/旧口头决定冲突时以 CHARTER 为准，Phase A 清单明写删 SealStamp。
2. **范围冻结 + 节奏优先**：印猫属"锦上添花"，落地它会把诗库/聊天页精修的连贯性切断，不值当。
3. **可逆**：删 import 零成本可回退；印猫决策只是"暂缓"不是"取消"，将来要做随时重开。

## 后果

**好的**：诗库页精修范围干净、无遗留 dead import；美术节奏不被 logo 工程打断。
**坏的**：印猫记忆点暂缺，登录页 logo 仍是藤字印。
**代价**：将来做印猫时需要重新引入组件 + 全站替换（favicon / 头像 / 加载态 / 成就时刻），是一次独立的小工程。

## 如果反悔了

- Demo 收尾（Phase E）阶段若觉得藤字印不够有辨识度，可重启印猫（案 C 朱砂印猫），作为独立视觉任务落地。
- 重启时注意：`SealStamp` 组件此时已无引用，需确认组件本身是否还在 `src/components/`，或直接重画。
