# 0007. 产品评审会:体验补全纳入 + 章程修订 #1

**日期**: 2026-07-06
**状态**: Accepted

## 背景

CHARTER 生效当天(Phase A 动工前),Owner 发起一次产品评审:以"资深用户"视角审视产品缺什么,以工程视角评估怎么落地,再逐项拍板要不要做。评审发现的核心问题:产品功能内核强(三层 Memory / grounding 出题 / LLM-as-judge),但**作为 App 的基础体验有硬缺口**——页面间无导航、根路径是脚手架空壳、无自动化测试。

按 CHARTER 第 9 条范围冻结,新想法默认进 backlog;本次经 Owner 拍板走章程修订流程,形成修订 #1。

## 决策(Owner 逐项拍板)

| # | 项 | 结论 | 归属 |
|---|---|---|---|
| A1 | 全局导航骨架(四入口:chat/诗库/错题本/我的) | ✅ 纳入 | Phase A |
| A2 | 根路径 `/` 重定向(未登录→/login,已登录→/chat) | ✅ 纳入 | Phase A |
| A3 | 对话历史列表(回看+续聊) | ✅ 纳入 | Phase A,可顺延 D |
| A4 | 性能优化包(字体子集化+诗库分页+Lighthouse) | ✅ 纳入 | Phase D · 5.3 P1 |
| B1 | BYOK 外接 API key(用户 key 优先,平台 key 回落) | ✅ 修订章程纳入 | Phase D · 5.3 P1 |
| B2 | 学习进度可视化 | ✅ 轻量版并入 | Phase C1 |
| B3 | 宣纸分享卡片 | ❌ 进 backlog | - |
| T | 测试路线 | ✅ 引入 Playwright smoke(新依赖已获批) | Phase D · 5.3 P1 |

## 关键理由

- **A1/A2 是体验硬伤而非新功能**:面试官点两下就会发现页面间没有导航、`/` 是 Next 默认页,归入"美术做透"的信息架构部分合理。
- **BYOK 的双重价值**:保护平台 key 成本(线上 demo 被刷=Owner 账单),且"key 加密存储 + 优先级路由 + 限流"是面试可讲的工程点。红线:key 不明文落库、不进日志。
- **Playwright**:Owner 在"手动走查清单(零依赖)"和"e2e(新依赖+1-2 天)"之间选了后者——招聘上"有 e2e"是加分项。依赖引入已按约定获 Owner 明确批准。
- **B2 并入 C1**:进度页与自适应出题是同一份画像数据的两个出口,并入后 C1 的"画像 → 出题参数"叙事有了用户可见的界面出口。

## 后果

**好的**:App 有了完整骨架;测试规范升级;BYOK 补上成本安全短板。
**坏的**:Phase A 和 Phase D 工作量各增约 1-2 天,时间盒压力上升。
**代价**:若进度吃紧,砍单顺序(CHARTER 第三节)照旧生效;A3 对话历史已预设可顺延 D 作为缓冲。

## 如果反悔了

- Phase A 若因体验补全超时 3 天,A3 先顺延、A1/A2 保住(它们是硬伤修复)。
- Phase D 若装不下 BYOK + 性能包 + Playwright,按"Playwright > 性能包 > BYOK"保序砍(测试规范 > 用户体验 > 锦上添花的工程点)。
