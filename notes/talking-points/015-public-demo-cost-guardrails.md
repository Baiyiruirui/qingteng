# 015 · 公开 Demo 的成本与滥用护栏

## 面试一句话

青藤不是只把 API 接上就公开部署：我把登录、LLM、embedding 和 ASR 的滥用面拆开，用现有 Upstash Redis 做用户/IP 双维度限流，再用输入边界控制单次最坏成本。

## 问题背景

作品集 Demo 必须公开可体验，但公开登录也意味着任何人都能调用付费模型。只限制 `/api/quiz/generate` 不够，因为聊天、开场白、主观题判题、语义搜索和朗读都各自连接第三方服务；攻击者还可以通过切换入口绕过单接口配额。

## 设计

### 两层预算

- 各入口有短窗口 burst limit，限制脚本瞬时并发。
- 所有 LLM/embedding 入口共享每用户与每 IP 的小时总预算，防止通过换接口规避。
- 腾讯 ASR 独立计量，因为它与 LLM 使用不同供应商和成本口径。

### 固定窗口键

Redis key 包含 scope、身份摘要和时间桶：

```text
qt:rl:<scope>:<sha256(identity)[0:24]>:<time-bucket>
```

`INCR` 提供原子计数，首个请求设置过期时间。原始用户 ID/IP 不写入限流键。响应包含 `Retry-After`，调用方知道多久后可重试。

### 失败策略

开发环境默认不启用，避免影响本地迭代；生产环境自动启用。Redis 故障时受保护请求 fail closed 返回 `503`，因为对公开付费接口而言，“暂时不可用”比“保护层失效后无限调用”更可控。

### 次数之外的输入边界

次数限流不能防住一条超大 prompt。服务端还会重新校验消息角色、条数、单条长度和总上下文；朗读接口按 Base64 实际长度复算音频大小；语义搜索同时限制 query 长度并在前端防抖。

## 为什么不用新依赖

项目已经依赖 `@upstash/redis` 支撑 Memory，限流直接复用它，减少一套服务和配置。当前固定窗口足够覆盖作品集 Demo；真正的大规模产品才需要滑动窗口、多地域一致性、设备指纹和风控平台。

## 可验证性

`pnpm verify:security` 会检查运行时环境、疑似明文密钥、Git 环境文件历史，以及高成本路由是否仍保有限流/门禁调用。它不会打印任何密钥值。

## 取舍

- IP 可能对应共享网络，所以注册限制保守设置为每小时 5 次。
- 已登录接口同时按用户和 IP 计数，兼顾单账号滥用与批量账号滥用。
- 当前没有 CAPTCHA 和每日金额熔断；第三方平台额度告警仍是最后一道保护。
