# 004. 学习画像的增量计算 + "背景知识而非台词"的注入哲学

**相关代码**: `src/ai/memory/mid-term.ts`, `src/ai/memory/build-context.ts`
**相关路由**: `src/app/api/chat/route.ts`, `src/app/api/conversations/route.ts`

## 一句话讲点

用字符串匹配从对话历史里低成本提取"聊过哪些诗/触及哪些主题",Redis 缓存避免每次重算;注入时把画像写成青藤的"背景记忆"而非"开场报告",只在话题相关时自然带出。

## 30 秒电梯版

每个用户有一份 `LearningProfile`(聊过的诗名、触及的主题、近 7 天活跃天数等)。计算方式:用 140 首诗的标题做字符串集合,在最近 40 条消息文本里做 `includes` 匹配,不调 LLM,秒级完成。算完缓存进 Redis(key: `user:{id}:profile`,TTL 1 小时)。

注入时拼在 `CHARACTER_SYSTEM_PROMPT` 后面,用"【仅供参考,不要主动罗列给他听】"框住,明确告诉模型这是内部背景,不是开场词。新对话创建时 invalidate 缓存,让下次重算带入最新数据。

## 2 分钟深度版

### 为什么不用 LLM 做诗名/主题提取

直觉方案:每次对话结束后让 LLM 分析"这次聊了什么诗"。问题:
- 每条消息多一次 LLM 调用,延迟 +1-2s,费用翻倍
- 提取结果本身也可能幻觉(LLM 认为"聊了《登高》"但其实只是随手提了一句)

字符串匹配的优势:
- 零延迟,零费用
- 精确:必须出现诗的标题才计入(漏报比误报好,漏报只是少记一首,误报会让青藤说出错误的"记忆")
- 140 首诗标题集合 in-memory,匹配是 O(N×M) 但 N=140、M=消息长度,微秒级

### 为什么画像是"背景知识而非台词"

用户要的是"被自然记得",不是"被汇报监控"。两种注入方式的差别:

**坏的方式(台词)**:
> 青藤:你好!我记得你聊过《登高》《春望》,触及孤独、思乡主题,近7天来过3天。

**好的方式(背景知识)**:
- 用户问"杜甫还写过什么" → 青藤自然说"嗯,你之前在《登高》上花了些时间……"
- 用户说"我今天心情差" → 青藤不说画像,只是共情——画像没有触发词

实现上:在 `build-context.ts` 里加明确的 prompt 约束"只在话题相关时自然带出,不要一开口就汇报这些信息,也不要因此增加提问的密度"。最后一句专门防止有了画像之后模型变得更爱反问。

### 缓存失效策略

不是每次消息后 invalidate(那样缓存永远是冷的),而是在"新对话创建"时 invalidate。理由:
- 新对话 = 自然的 session boundary,这时用户刚结束上一轮,数据最值得更新
- 单次对话内,画像不需要实时更新——青藤已经知道"你聊过登高",这个事实不会在对话中途改变
- TTL 1 小时作为兜底,保证缓存不会永远不更新

### 降级策略

`getProfile` 用 try/catch 包整个逻辑。Redis 或 DB 任意一层出故障,返回 `null`,`buildSystemContext` 遇到 `null` 返回 `''`,chat route 用 `CHARACTER_SYSTEM_PROMPT + ''` = 原版 prompt。降级对用户无感,只是没有画像背景,退化到"无记忆模式"。

## 可能的追问

- Q: 字符串匹配会漏报吗?比如用户说"老杜"没说"杜甫"?
  A: 会漏。匹配是针对诗的**标题**,不是作者名。"《登高》"会命中,"老杜的那首"不会。这是 MVP 的合理妥协——宁可少记不要乱记。

- Q: 主题(themes)从哪来?
  A: 从 `poems.themes` 字段——导入 140 首诗时就有了,比如《登高》的 themes 可能是 `["羁旅","忧国","孤独"]`。命中某首诗 → 把这首诗的 themes 全部收入 matchedThemes。

- Q: 为什么在 `conversations` 路由 invalidate 而不是在 `chat` 路由的 onFinish 里?
  A: 每条消息 onFinish 后 invalidate 意味着每次都走冷计算路径(3-4 次 DB 查询),把缓存的意义抵消了。"新对话创建"是更合适的 invalidation 点:一次对话内数据不需要实时更新,对话结束后刷新最合适。
