# 016. 对话可靠性与用户可控 Memory

**相关代码**: `src/app/api/chat/route.ts`、`src/lib/chat-reliability.ts`、`src/app/api/memories/`

## 一句话讲点

青藤不只“记得用户”，还保证记忆来源可信、消息可恢复，并把查看、纠正和删除权交还给用户。

## 30 秒电梯版

聊天系统现在只接受本轮用户消息，完整历史由服务端按会话所有权从数据库重建。每轮消息有稳定幂等键，重复请求不会重复生成；流后落库失败会进入 Redis 恢复队列。Memory 侧提供查看、纠正、删除、暂停和保留期限设置，并在写入短期、中期、长期记忆前统一检查策略。

## 工程权衡

- 不改数据库 schema：幂等键暂存于消息 meta，并由 Redis 锁覆盖公开 Demo 的并发场景。
- 不把 Memory 控制只做成前端开关：暂停状态保存在 Redis，服务端抽取与各层写入都会检查。
- 不把“删除”做成隐藏：列表、单条修正、单条删除和全部清理都有用户可见入口。
- 不承诺声学级发音评分：腾讯 ASR 只用于转写和字句对齐，界面使用“朗读掌握度”而非“发音分”。

## 可能的追问

- Q: Redis 故障时幂等还能绝对保证吗?
  A: 当前没有为消息增加数据库唯一约束，因此极端 Redis 分区下仍有重复写入的残余风险。作品集阶段选择零迁移方案；正式多实例生产应增加数据库级唯一键作为最终防线。

- Q: 用户暂停 Memory 后，旧记忆怎么办?
  A: 暂停阻止后续抽取和写入，旧记忆仍可逐条纠正或删除，也可以全部清理。保留期限由服务端在读取和维护时执行。

- Q: 为什么拼音来自 TTS 字幕而不是本地词典?
  A: 它能与示范音保持同源并避免引入新依赖。仅在腾讯返回完整音素序列时展示，数据不完整就退化为只有示范音。
