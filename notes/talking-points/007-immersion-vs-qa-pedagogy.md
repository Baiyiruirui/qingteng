# 007. 诗境沉浸 vs 问答:产品差异化的教育学根基

**相关代码**: `src/app/api/session/`（roleplay 模式，Week 3 Day 2 实现）、`src/ai/prompts/v1/immersion-memory.ts`
**相关数据**: `data/immersion-scripts.json`、`immersion_scripts` 表
**相关决策**: `notes/decisions/0004-conversation-mode-architecture.md`

## 一句话讲点

市面上的古诗词产品都在做"讲解"（电子教辅），青藤做的是"体验"——基于建构主义和体验式学习理论，让学生先成为诗中人、先生成自己的情感，再与作者共鸣。这是 LLM 角色扮演能力能做、传统教辅做不到的事。

## 核心区别

| 维度 | 问答 / 讲解模式 | 诗境沉浸模式 |
|---|---|---|
| 学生的位置 | 旁观者（"这首诗讲了什么"） | 当事人（"你是李白，你此刻……"） |
| 情感来源 | 理解作者的情感 | 先生成自己的情感，再对照作者 |
| 记忆留存 | 听过就忘（被动接收） | 亲历的记得牢（主动建构） |
| "低头思故乡" | 老师告诉你"这表达思乡" | 你被引导着自己说出"想起了家"——那一刻这句诗成了你的 |

## 理论根基

- **建构主义（Constructivism）**：知识不是被灌输的，是学习者在情境中自己建构的。
- **体验式学习（Experiential Learning, Kolb）**：具体经验 → 反思观察 → 抽象概念化 → 主动实践的循环。沉浸模式让学生从"具体经验"（扮演诗中人）入手，而非直接给"抽象概念"（诗的主旨）。

古诗词教学最难的不是字面意思（查字典即可），而是**情感共鸣**。讲解传递不了共鸣，体验才能。这正是青藤选择"让 LLM 做受控角色扮演"而非"让 LLM 做更好的讲解"的根本原因。

## 30 秒电梯版

市面上的古诗词 App 本质都是电子教辅——把诗的意思、手法、背景讲给学生听。我做的是体验：基于建构主义，让学生先成为诗中人、先长出自己的情感，再发现千年前有人与自己共鸣。古诗词最难的情感共鸣只能靠体验达成，讲解达不到。这是我用 LLM 角色扮演能力实现的差异化。

## 2 分钟深度版

讲三层：
1. **问题**：古诗词学习的真正难点是情感共鸣，不是字面理解。但传统产品（包括大多数 AI 套壳）都在优化"讲解"，越讲越细，却始终是单向灌输。
2. **方案**：用 LLM 的角色扮演能力，把学生放进诗的情境里当事人。关键是这不是开放式聊天——每首诗有预设的教学目标（teachingGoals）和情感节点（keyBeats），青藤是"受控地"引导，既保留角色扮演的代入感，又确保始终朝教学目标推进。
3. **工程实现**：脚本设计成"引导地图"而非"台词"——只给目标和节点，不写死对白，让 LLM 根据学生真实回应灵活推进。这是"受控的开放式角色扮演"，平衡了沉浸感与教学可控性。

### Day 2 实现细节（受控角色扮演机制）

**脚本即引导地图**：`immersion_scripts` 表存储每首诗的 `teachingGoals`（2-3 条教学目标）、`keyBeats`（按顺序的情感节点路线图）、`exitCondition`（收束条件）。青藤拿到这份地图，不照读台词，而是根据学生真实回应即兴推进——学生说 A 就接 A，学生说 B 就接 B，但始终朝 teachingGoals 推。

**teachingGoals 锚定**：system prompt 明确告诉青藤"这次沉浸要让学生体会到的几件事"，配合"不要说'现在我们来体会第二个目标'"这类约束，保证教学结构对学生不可见——学生以为在自然对话，实际上被温柔地引导向情感核心。

**跑题拉回机制**：prompt 明确约束"学生跑题或敷衍时，温和地把他拉回情境，不要生硬"。实测：学生说无关内容时，青藤会用"你刚才说……让我想起你此刻站在这里……"的方式将话题拉回诗境，既承接了学生的发言，又回到了引导线。

**不注入三层 Memory,但出境后抽取 Memory**：沉浸模式 system prompt 只注入诗境脚本，不注入 short-term / mid-term / long-term memory。原因：沉浸是情境专注的，用户上一次说"喜欢李白"不应该打断这次《登高》里杜甫的情境。但沉浸结束后,`extractImmersionAndStore()` 会用专门 prompt 判断学生是否表达了真实情绪、偏好或困惑,只把"这位学生在《登高》的沉浸对话里说出了自己的孤独感"这类信号写入长期 Memory,不把角色扮演剧情当成真人经历。

## 可能的追问

- Q: 怎么保证角色扮演不跑偏（学生乱答、青藤被带跑）？
  A: 三层保险：① system prompt 里 teachingGoals 作为目标锚点；② keyBeats 提供按顺序的推进路线，LLM 知道下一步该把学生引向哪；③ prompt 明确"学生跑题时温和拉回"。实测：学生说"我以为是霜，原来是月光"，青藤立刻接："月光太亮，亮到让你恍惚——你心里是不是本就带着一丝凉意？"——既接住了回应，又推进到 keyBeat 1。

- Q: 这种体验式教学有数据证明效果更好吗？
  A: 建构主义和体验式学习是成熟的教育学理论，有大量实证支持。青藤的产品假设是把这套理论用 LLM 落地。真实效果验证需要 A/B 测试和留存数据，是产品上线后的事。

- Q: 为什么不让 140 首诗都支持沉浸？
  A: 沉浸脚本需要文学性的人工设计（角色、场景、教学目标、情感节点），不能机械生成。精雕 3 首做范式验证，远胜粗制 140 首。规模化时可以做"脚本生成 + 人工审核"的半自动 pipeline。

- Q: 沉浸模式为什么不直接用用户 Memory?
  A: 入境时不注入,出境后再抽取。入境时保持诗境纯净,避免历史偏好打断当前角色扮演;出境后把真实学习信号沉淀回长期 Memory。这样既保留沉浸感,又不丢掉最有价值的情绪和困惑信号。
